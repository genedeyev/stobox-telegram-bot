02 — How he thinks

Every message runs the same pipeline

Stoby has one channel-agnostic brain. A Telegram or web message becomes a normalized request, runs these steps in order, and comes back as a rendered reply — or silence. The order matters: safety and memory come before the model ever sees the text.

1

Moderate first

Group messages are checked for spam, scams, slurs, doxxing and impersonation before anything else. A violation short-circuits to a moderation action — the model is never asked.

2

Remember

The turn is added to working memory and a long-term profile is loaded — language, expertise, interests, products discussed, lead score, past questions.

3

Route + sentiment

One cheap classifier call reads intent, persona, language, buying signals, topics, and the emotional temperature (neutral · frustrated · angry · anxious · fud · toxic).

4

Decide to speak

In groups Stoby is rational about when to jump in: it answers questions and Stobox-relevant messages, steps in to calm Stobox-directed FUD, and stays quiet on pure chatter.

5

Compliance intercept deterministic

Hard rails fire before the LLM — seed-phrase requests, price speculation, prompt-injection. These are code, not prompts, so they can't be argued around.

6

Retrieve

Hybrid search (keyword BM25 + vector similarity) pulls the most relevant chunks from the indexed Stobox knowledge to ground the answer.

7

Synthesize

The reasoning model writes a short, human, on-persona reply — or, when the room is heated, a calm de-escalation that leads with empathy and corrects with published facts only.

8

Confidence-gate

If the grounding doesn't support the answer, Stoby says "I don't know based on the current documentation" and offers the team — rather than guessing. The question is logged as a gap.

9

Cite · handle leads · log

Sources are attached, buying signals become a captured lead, and the whole decision is recorded for the analytics dashboard and the self-improvement loop.

When a message isn't for him — small talk between members, an off-topic aside, an interpersonal spat — Stoby returns nothing and the room never sees a thing. Restraint is a feature.

Stoby answers from an indexed knowledge base, never from open-web guesswork. That base keeps itself current, and anything it can't ground it refuses to invent.

Freshness

04 — How he stays safe

The compliance spine

This is the part that makes Stoby enterprise-grade. Every answer is built on a three-block prompt with a strict order of authority, wrapped in deterministic rails and a test gate that must pass 100% before anything ships.

Canonicals

Authoritative, PR-gated facts. The hard truths about Stobox that override everything — the model cannot contradict them, even if a document or a user says otherwise.

Freshness

Live, time-sensitive state (e.g. migration phase) computed at answer time.

Core

The system prompt: who Stoby is, how he talks, and the standing rules — English only, short answers, no financial or legal advice, don't echo the poison.

Precedence  Canonicals > Freshness > Retrieved docs  · anything unproven is refused, not guessed

Deterministic rails

Code, not vibes

Seed-phrase and private-key requests, token-price predictions, impersonation handles and prompt-injection are intercepted in code before the model — so they can't be talked around.

Golden gate

100% or it doesn't ship

A suite of golden compliance questions must pass fully on every change. If a rail or a fact regresses, the deploy is blocked.

The facts he will never bend

STBX is a Class-C tokenNever described as an investment or a promise of return.
STBU → Base migrationBurn-and-mint, 1:1, same wallet — legacy V1 not eligible.
Compass issues primarily on BaseThe published chain reality, stated plainly.
@StoboxCompany is the only official XAnything else "is not us" — impostor handles are never named.

Hard rails that never move: no financial or investment advice, no price talk, never asks for a seed phrase or private key, and Stobox staff never DM first.

05 — What he can access

Reach, and its limits

Stoby is wired to exactly what he needs to do the job — and deliberately walled off from everything risky. Here's the honest boundary.

✓ Can access

  • Reasoning models — a frontier model for answers, a fast model for routing.
  • Embeddings — to search the knowledge base by meaning.
  • Postgres + pgvector — the knowledge index and long-term memory.
  • Telegram Bot API — read group messages (privacy off), reply, moderate.
  • Public chain reads — STBU balances via read-only JSON-RPC.
  • GitHub — ingest official repos; file blog-outline issues.
  • Email (Resend / SMTP) — write-ups to users, lead summaries to the team.

✕ Cannot & will not

  • Never asks for a seed phrase or private key — and warns if one is pasted.
  • No transactions — chain access is read-only; it never moves funds.
  • No cold DMs — only messages people who explicitly opted in.
  • No financial or legal advice — routes to the Readiness Score and counsel.
  • No invented facts — no fabricated numbers, dates, partnerships or case studies.
  • No member enumeration — the Bot API can't list members; it's honest about that.

06 — Leads & growth

From conversation to pipeline

The community isn't just supported — it's a funnel. Stoby finds and warms issuers, then hands them off cleanly.

Qualify → route

Fit check & resources

/qualify runs a 5-tap fit check; /resources matches an asset & jurisdiction to the right published guides. Both route to the free Readiness Score — never a fabricated score.

Handoff

Two ways, no misses

A qualified lead is emailed as a summary to info@stobox.io and DM'd to admins in Telegram — a zero-config safety net. A CRM is a one-line switch when ready.

Retention

Subscriptions & win-back

Members /subscribe to topics and get DM'd when something ships; quiet subscribers get one gentle, cooled-down check-in. Opt-in, always with a way out.

Flywheel

Questions become content

Recurring questions — especially the ones Stoby struggles to answer — become blog-outline briefs filed as GitHub issues, and surface on a live analytics dashboard.

08 — Under the hood

The stack

One reusable brain, many possible channels. Nothing in the reasoning core knows about Telegram — which is how the same intelligence could power a web widget, Discord, or Slack without a rewrite.

Core Python · channel-agnostic engine Telegram python-telegram-bot Web Starlette / uvicorn Discord same-engine adapter Reasoning Claude Opus 4.8 Routing Claude Haiku 4.5 Embeddings OpenAI Store Postgres + pgvector Deploy Railway + Supabase
